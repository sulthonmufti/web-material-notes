Git: How It Works

Membedah Mekanisme Snapshot, Hash, dan Data Integrity

1. Snapshots, Bukan Deltas +

Mayoritas sistem VCS lain (seperti SVN) menyimpan data sebagai Deltas (daftar perubahan file dari waktu ke waktu). Git berbeda.

Git menganggap datanya sebagai serangkaian potret (snapshot) dari miniatur sistem file. Setiap kali Anda commit, Git memotret semua file Anda saat itu dan menyimpan referensi ke potret tersebut.

ğŸ¢ Case Industri: Kecepatan Checkout

Karena Git menyimpan snapshot, saat Anda pindah branch (checkout), Git tidak perlu menghitung ulang perubahan file satu per satu. Ia tinggal mengganti file di folder Anda dengan versi yang ada di snapshot. Inilah kenapa Git sangat cepat meski proyeknya besar.

2. SHA-1: Identitas Tak Terbantahkan +

Bagaimana Git tahu jika sebuah file berubah meski hanya satu spasi? Jawabannya adalah Hashing.

ğŸ“– Istilah: SHA-1 Hash

String 40 karakter (contoh: 24b9da...) yang dihasilkan dari isi file atau struktur direktori. Jika isi file berubah 1 bit saja, Hash-nya akan berubah total.

Simulasi Mekanisme Hash:

Hash ID: e04fd...

Di Git, Anda tidak bisa mengubah isi file tanpa Git mengetahuinya. Ini disebut Data Integrity.

3. Alur Kerja: The Three States +

Ini adalah siklus hidup file yang paling sering digunakan dalam industri setiap harinya.

  • Committed: Data sudah tersimpan aman di database lokal (Snapshot).
  • Modified: Anda sudah mengubah file tapi belum menyimpannya ke database.
  • Staged: Anda menandai file yang dimodifikasi untuk masuk ke snapshot berikutnya.

4. Database Lokal: Folder .git +

Saat Anda menjalankan git init, sebuah folder tersembunyi bernama .git dibuat. Inilah "otak" dari proyek Anda.

ğŸ¢ Case Industri: Backup & Portability

Karena semua "How it works" disimpan di dalam folder .git, Anda bisa meng-copy folder proyek Anda ke flashdisk, dan seluruh riwayat (history) akan ikut terbawa. Git tidak butuh koneksi server untuk melihat log masa lalu.
